# AI Usage Widget

Native iOS WidgetKit app plus a Mac connector for showing Claude and Codex usage/status on an iPhone widget.

Canonical planning/status files:

- `outputs/AI-Usage-Widget-Implementation-Plan-v4.md`
- `outputs/AI-Usage-Widget-Implementation-Status.md`
- `CONTRACT.md`

## Fresh Local Setup

```sh
npm run setup:local
```

This is the safest first command on a fresh Mac checkout. It installs missing root and connector npm dependencies, creates `ios/Config/Local.xcconfig` from the example if needed, generates the Xcode project if needed, builds the connector, verifies the packaged `aiuw` CLI path, and runs the no-signing local device preflight.

It does not install the LaunchAgent, create or rotate pairing tokens, start a connector server, change Apple signing, or touch iPhone state. Use `npm run setup:local:check` when you want the same readiness check without writing missing setup files.

## Source Hygiene

Authoritative source lives in `src`, `connector/src`, `ios/Shared`, `ios/AIUsageWidgetApp`, `ios/AIUsageWidgetExtension`, `ios/AIUsageWidgetTests`, `ios/Tools`, `fixtures`, `tests`, `scripts`, `CONTRACT.md`, `ios/project.yml`, and the status/evidence docs under `outputs`.

Generated, vendor, or local proof artifacts include `node_modules`, `connector/node_modules`, `connector/dist`, `build`, `.swift-module-cache`, `ios/AIUsageWidget.xcodeproj`, `work/XcodeGen`, `outputs/widget-previews`, `outputs/device-evidence/00-*.json`, and `outputs/device-evidence/screenshot-proofs.json`. Do not review those as product source unless you are specifically checking generated proof output. `outputs/device-evidence/manifest.json` is the stable screenshot proof spec and stays source-like even though other files in that folder are generated by device checks. `npm run check:source-hygiene` verifies this boundary and is included in `npm run check:scripts`.

`npm run check` validates the TypeScript contract, connector tests, iOS plists/entitlements, Swift parsing, all six fixtures, embedded Swift preview fixture parity with root fixtures, pairing host rules, executed stdin pairing-validator wrapper behavior, executed oversized paste/token rejection, executed manual terminal-block pairing paste, executed redacted-payload rejection with the manual-pairing message, unavailable `capturedAt` rejection, unsafe raw snapshot rejection, mocked iOS LAN request behavior with sanitized diagnostics, the widget refresh coordinator's no-device branches, the separate approval-request/approval-decision channel, bounded local approval-attempt diagnostics, fresh unsigned WidgetKit AppIntent metadata proof, fresh native XCTest build proof, strict physical-iPhone discovery false-positive fixtures, the same-Wi-Fi mock connector local-pairing proof, the signing-free Mac-owned approval loop proof, the generated LaunchAgent ProgramArguments smoke, deterministic fake CodexBar subprocess proof, the no-signing local device preflight, and the local-only evidence audit.

`npm run check:connector-package` verifies the connector package entrypoint stays on the built `aiuw` CLI path in both `connector/package.json` and `connector/package-lock.json`, and confirms the built CLI still has its node shebang.

`npm run check:ios-config` is a no-signing App Group/config drift guard. It verifies `Local.xcconfig`, `Local.xcconfig.example`, XcodeGen source wiring, source plists/entitlements, QR scanner camera permission text, privacy manifests, and any existing simulator/iphoneos build products all agree on `APP_GROUP_IDENTIFIER`; an empty `DEVELOPMENT_TEAM` is only a warning here.

`npm run check:appintents` verifies Xcode's extracted WidgetKit/AppIntent metadata from the existing unsigned `Debug-iphoneos` build. `npm run check:appintents:fresh` first checks whether that metadata is already fresh, rebuilds with `npm run device:build:unsigned` equivalent only when it is missing/stale/invalid, and then validates again. Root `npm run check` uses that fresh path so stale build metadata cannot pass. The check reads both the standalone widget extension product and the widget embedded inside the app, then fails unless the built metadata contains exactly `RefreshUsageIntent` with no parameters plus `ApprovalDecisionIntent` with only `requestId` and `actionRawValue`; both must keep `openAppWhenRun=false` and no interactive/freeform text-like input.

`npm run check:product` is a no-device product-invariant guard. It fails if runtime/source files drift from `usage` back to allowance/额度 wording, contract/plan docs show invalid unavailable usage placeholders, user-cacheable fixtures grow fake percentages, the fixed widget header changes, runtime copy drifts from the sharper mocking emoji direction back to rejected gentle fallback emoji or the old `👉` pointer, fixtures stop being exactly Claude + Codex, Hermes/OpenClaw leak into the V1 app/widget provider surface, App Group fallback config is silently accepted, app/widget approval can drift from Mac-owned matching-receipt semantics, persisted widget/approval diagnostics start storing hostnames, URLs, bodies, or tokens, app-side LAN troubleshooting or saved-pairing display drifts back to raw host display instead of safe address indexes/counts, widget LAN refresh loses its total refresh budget or drifts away from the shared coordinator, or widget LAN/approval host attempts drift from the pairing schema cap.

`npm run smoke:connector` starts a temporary mock connector, fetches ping/snapshot, rotates the pairing token while the server is still running, verifies the old token fails, and verifies the new token works.

`npm run smoke:local-pairing` is the signing-free same-Wi-Fi proof. It starts a temporary mock connector, rotates a manual full-token pairing payload, validates that payload with the same Swift parser used by the app, then pings and fetches a Swift-validated snapshot from at least one advertised iOS-accepted address without printing the bearer token. Failed fallback attempts are summarized by count only, and they do not fail the proof as long as one address works, matching the iPhone pairing fallback model.

`npm run smoke:local-approval` is the signing-free Mac-owned approval proof. It starts a temporary mock connector in an isolated config dir, creates a bounded pending request, fetches it through `GET /v1/approval-request`, posts a minimal approve decision through `POST /v1/approval-decisions`, proves the connector clears only after a matching request-id/action receipt, validates the approval responses through the shared Swift decoder, and checks the command outputs do not leak the bearer token.

`npm run smoke:service-program` is the signing-free LaunchAgent command proof. It generates the real `aiuw service plist`, parses its `ProgramArguments`, starts that exact `node ... cli.js start --quiet-pairing` command with an isolated config dir and random port, then verifies authenticated `/v1/ping`, `/v1/snapshot`, `aiuw status --live`, and `aiuw doctor --live` without installing or loading launchd.

`npm run smoke:real-collectors` runs the safe Tier 0 Codex/Claude collectors on this Mac with a stripped shell path, validates the sanitized snapshot shape, and confirms the connector does not emit usage `capturedAt` or percentages without an explicit usage source.

The real connector start path also tries the local Codex CLI app-server usage source for Codex. It calls `codex app-server` over stdio, asks only for `account/rateLimits/read`, and copies only the 300-minute and 10,080-minute `usedPercent/resetsAt` windows into the snapshot as remaining percentages. If that probe fails, Codex falls back to `no_stable_source` instead of showing a fake meter.

`npm run smoke:codexbar:fake` is the deterministic optional-M4 adapter proof included in `npm run check`. It creates a temporary fake `codexbar` executable, runs it through the same limited subprocess path as a real adapter, feeds poisoned stdout/stderr plus unsafe JSON fields, and prints only sanitized Claude/Codex usage summaries. It does not require or call a real CodexBar install.

`npm run connector:doctor` is the read-only Mac connector setup summary. It reports whether a pairing config exists, whether the LaunchAgent plist is installed/loaded/running, whether the installed LaunchAgent's Node/CLI paths still exist, and the next setup actions without printing tokens, snapshots, provider output, raw launchctl output, or response bodies. `npm run connector:doctor:live` additionally performs the authenticated `/v1/ping` plus `/v1/snapshot` live check when a pairing config exists.

`npm run connector:service:status` is a read-only LaunchAgent check. It tells you whether the run-at-login connector plist exists and whether launchd reports it loaded/running; it does not prove the authenticated connector endpoints are reachable.

`npm run device:preflight:local` is the no-signing version of the device preflight. It keeps the App Group, privacy, Local Network, QR payload, Swift parser, connector, QR address candidate validation, and mock QR snapshot reachability checks as hard local gates, while downgrading only empty `DEVELOPMENT_TEAM` and physical iPhone visibility to warnings.

`npm run device:preflight` is the real-device diagnostic. It is expected to fail until signing values are filled and Xcode can see the iPhone; use it before the iPhone test loop, not as a CI gate. It also checks camera/privacy/local-network source wiring, starts a temporary mock connector with quiet pairing output, checks advertised QR address candidates from this Mac by count, validates the full-token QR payload with the same Swift parser used by the app, validates mock `/v1/snapshot` through the Swift snapshot decoder, and reports LaunchAgent status as a warning-level run-at-login hint.

`npm run device:signing` is a read-only signing/device diagnosis that does not write project state. It checks `Local.xcconfig`, Xcode first-launch state, local Apple Development identities, `DEVELOPMENT_TEAM` identity match when possible, local provisioning profile presence, and iPhone visibility through the shared strict CoreDevice/Xcode parser. Failed `devicectl` JSON, unavailable/remembered/untrusted devices, and simulators do not count as a ready iPhone. Use it when `DEVELOPMENT_TEAM` or device install is blocked.

`npm run device:signing:apply` safely fills only the `DEVELOPMENT_TEAM` line in `ios/Config/Local.xcconfig` after Xcode has created/downloaded Apple Development identities. It applies automatically only when one Team ID is inferable; use `npm run device:signing:apply -- --team TEAMID` when multiple teams are present, and `--force` only to intentionally replace a different existing value.

`npm run device:evidence` is the post-run proof checklist. It is token-safe: it checks signing/project/build/connector evidence that can be verified from the Mac, source and bundled camera/local-network/privacy values, distinguishes unsigned iphoneos compile proof from a signed installable device product, validates `outputs/device-evidence/manifest.json`, checks any saved proof screenshots are usable PNGs, and binds present screenshots to `outputs/device-evidence/screenshot-proofs.json` by manifest index, proof-text hash, file hash, byte size, dimensions, and modified time. After adding or replacing screenshot PNGs, run it once to write the updated sidecar and run it again to prove the screenshots still match. It writes `outputs/device-evidence/00-evidence-summary.json` with bounded pass/warning/failure counts and screenshot proof counts, without raw command output, local host details, device identifiers, pairing payloads, bearer tokens, or screenshot contents. Before install proof exists, missing screenshots remain warnings so signing/install blockers stay clear. After install proof exists, strict `npm run device:evidence` exits nonzero until all required screenshots are present and sidecar-bound.

`npm run device:evidence:local` runs the same token-safe audit with `--local-only`. It keeps source/build/App Group/URL-scheme/privacy/AppIntent metadata/screenshot-manifest failures hard, but downgrades the missing Apple signing team to a warning while signing is still blocked. It is not installed-iPhone proof, WidgetKit-rendering proof, iOS Local Network permission proof, or phone-to-Mac approval proof; strict `npm run device:evidence` remains the post-install gate.

`npm run device:runbook` prints the token-safe iPhone proof runbook from `outputs/device-evidence/manifest.json`, including the exact screenshot filenames, the Mac-owned approval loop, and the two-pass `device:evidence` sidecar binding flow.

`npm run device:build` builds the signed `Debug-iphoneos` app and embedded widget after signing is configured, then verifies the app/widget signatures, embedded provisioning profile, and App Group signed entitlements. `npm run device:install` builds and then installs with `xcrun devicectl`; it runs the same installability gate first, including when `--skip-build` is used, and uses the shared strict device-discovery parser to locate one available physical iPhone. It does not start the connector, rotate pairing tokens, or touch launchd. Use `npm run device:install -- --allow-provisioning-updates` only if you intentionally want Xcode to update signing profiles or register the device.

`npm run build:ios:tests` builds the native XCTest bundle for the generated Xcode project against generic `iphoneos` with signing disabled. `npm run check:ios-tests:fresh` is the read-only freshness guard for that proof; it fails when the built `AIUsageWidgetTests.xctest` or `AIUsageWidgetTests_iphoneos*.xctestrun` is missing or older than relevant iOS source/config inputs. The tests cover the Swift snapshot contract, rejection of `allowance` fields, unsafe raw field rejection, App Group identifier resolution, local-only pairing, bounded host lists, manual terminal-block pairing paste, redacted-payload rejection, quick approval fetch/storage/submission, mismatched Mac approval receipt rejection, bounded approval-attempt diagnostics, the shared approval decision handler used by the app and widget AppIntent, and token-free widget/LAN diagnostics including sanitized transport errors. It also compiles shared helpers such as the widget refresh coordinator through the generated app/widget/test source phases. It is a separate explicit Xcode compile/build proof; running the tests still needs a healthy simulator or real device runtime.

## Mock Connector

Start the local same-Wi-Fi mock connector:

```sh
npm run start:mock
```

The connector prints a QR code and a redacted JSON summary. The iPhone app should normally pair by scanning the QR code.

By default the terminal JSON is redacted and is not pasteable. The QR still contains the token. If camera scanning fails and you need manual paste, run:

```sh
npm run pair:manual
```

Manual pairing rotates the local connector bearer token. After running it, paste the new payload into the iPhone app and use that new pairing; older QR codes or saved pairings will return `401`. The app can accept either the raw JSON or the bounded terminal block that contains the `--manual` JSON, but redacted terminal output is still rejected because it has no real token.

## Real Tier 0 Connector

Start the connector against safe local CLI status collectors:

```sh
npm run start:real
```

This path only collects Codex/Claude CLI presence, version, auth/status, and coarse health signals. It tries the command from `PATH` first, then common fixed local CLI paths such as `~/.local/bin`, `~/.hermes/node/bin`, `/opt/homebrew/bin`, and `/usr/local/bin` so Finder/launchd-style environments are less brittle. It can also discover executable Claude Code binaries in `~/.local/share/claude/versions` and the bundled Codex CLI at `/Applications/Codex.app/Contents/Resources/codex`. If CLI version output is unavailable, it may read only `CFBundleShortVersionString` from `/Applications/Codex.app` or `/Applications/Claude.app` as sanitized version metadata. Codex can also run `codex doctor --json --summary`, but the connector reads only the redacted `overallStatus` and keeps all detail fields local. App presence never proves login or usage. It can classify explicit Claude third-party/provider-route signals as `third_party_no_quota`, while official Anthropic base URLs remain first-party unknown usage. It does not read cookies, Keychain values, transcripts, session logs, or provider private storage, and it does not produce usage percentages. If exact usage cannot be proven, the snapshot uses `usage.available=false`.

Use this smoke check before relying on real local collectors:

```sh
npm run smoke:real-collectors
```

## Quick Approval Channel

The app and medium widget can show a bounded pending approval request with only two actions: approve or deny. There is no message/comment input; users should open Claude/GPT directly if they need to type.

The Mac connector owns the pending request:

```json
{"ok":true,"request":{"schemaVersion":1,"id":"fixture-approval","source":"Mac 请求","title":"允许这一步继续？","createdAt":"2026-06-11T00:00:00.000Z","expiresAt":"2026-06-11T00:15:00.000Z"}}
```

The app's `从 Mac 拉请求` action and the widget's best-effort LAN refresh fetch this through `GET /v1/approval-request`. The connector authenticates the request and validates the pending request again before returning it. A successful `request:null` means the Mac has no new request to offer; it does not clear a local Mac-owned pending request. Network failure also leaves the current pending request visible instead of pretending the Mac accepted anything.

When a paired Mac exists, app/widget approval first submits a minimal authenticated payload to the connector. The app buttons and widget AppIntent use the same shared decision handler that is covered by the no-device Swift validator:

```json
{"schemaVersion":1,"requestId":"fixture-approval","action":"approve"}
```

The connector accepts this through `POST /v1/approval-decisions`, stores a local private JSONL decision record, clears a matching Mac-side pending request, and keeps `/v1/snapshot` unchanged. iOS only clears a Mac-owned request after the connector returns `ok:true` with the same `requestId` and `action` that was submitted. `npm run connector:approvals` shows a bounded read-only summary with request id, action, timestamp, and pending request metadata only. If the Mac is unreachable or returns a mismatched receipt, the iOS side leaves the request pending instead of pretending it worked. App buttons and widget taps both write one local App Group diagnostic that the app's `快速批准` section can show for device QA; that record stores only outcome, short detail, optional HTTP status, and host-attempt count, never tokens, URLs, bodies, prompts, comments, or raw logs.

Without pairing, only the app-created local UI fixture can be approved locally. That fixture is deliberately labeled `source="本地测试"` with an id starting `local-fixture-`; Mac-owned requests never clear through the local-only path and must reach the paired Mac before the app or widget records the decision.

For local Mac-loop testing after the connector has been built, these direct CLI actions seed and clear the pending request:

```sh
npm run smoke:local-approval
npm run connector:approvals:seed
npm run connector:approvals
npm run connector:approvals:clear
```

## Optional CodexBar Usage Adapter

Probe CodexBar without starting the connector:

```sh
npm run smoke:codexbar
```

The live probe is manual because CodexBar is optional local tooling. It is safe to run when CodexBar is not installed: it reports `skipped: true` and `reason: "codexbar_not_found"` instead of failing the fixture-first build. When CodexBar is installed, it still prints only command status plus the allowlisted usage summary. It never prints raw CodexBar stdout/stderr. The deterministic `npm run smoke:codexbar:fake` proof is the one included in `npm run check`.

Start the connector with explicit CodexBar usage parsing:

```sh
npm run start:codexbar
```

This is opt-in because CodexBar has its own local config/cache access. The connector invokes `codexbar` with fixed argv, no shell, output caps, and timeouts, then copies only exact normalized `claude`/`codex` provider names, sanitized version, `usage.primary.usedPercent/resetsAt`, and `usage.secondary.usedPercent/resetsAt` into our snapshot. Aliases, future names, and proxy names such as `Claude Code`, `OpenAI Codex`, OpenClaw, Hermes, or Claude proxy are ignored until explicitly approved. CodexBar is a usage overlay only: it can fill usage only for a connected, working Tier 0 provider whose usage reason is still `no_stable_source`; it cannot override logged-out, third-party/no-quota, degraded, error, or already-metered states. Account emails, identities, dashboard details, tokens, non-version text, and raw CodexBar output are dropped.

Check connector config without starting the server:

```sh
npm run connector:status
```

`aiuw status` is a read-only pairing/config summary. It shows whether pairing config is present, whether Mac identity fields are configured, the port, advertised address count, and the explicit manual-pair command; if no pairing config exists, it reports that without creating one. It does not print raw LAN hosts, config paths, Mac device ids/names, tokens, or redacted pairing payloads, and it does not prove a connector server is currently running.

For a one-shot setup diagnosis, run:

```sh
npm run connector:doctor
npm run connector:doctor:live
```

`aiuw doctor` summarizes pairing config, LaunchAgent state, optional live connector readiness, and next actions in token-free JSON. The live variant is useful after starting `npm run start:mock`, `npm run start:real`, or a loaded LaunchAgent.

When the connector is expected to be running, use the live probe:

```sh
npm run connector:status:live
```

`aiuw status --live` calls authenticated `/v1/ping` plus `/v1/snapshot` on the advertised local addresses and reports address counts, `snapshotLooksValid`, and sanitized failure categories. It does not print the bearer token, authorization header, raw host list, raw exception text, snapshots, provider output, or raw response bodies. This proves Mac-local reachability/auth/snapshot shape only; the iPhone still needs the Local Network permission and same-Wi-Fi test.

## Run Connector At Login

Generate the user LaunchAgent plist without writing files:

```sh
npm run connector:service:plist
```

Before installing it, you can prove the generated command path boots and serves live status without writing to `~/Library/LaunchAgents`:

```sh
npm run smoke:service-program
```

Install the plist into `~/Library/LaunchAgents`:

```sh
npm run connector:service:install
```

This only writes the plist; it does not call `launchctl`. The installed plist runs the connector with `--quiet-pairing`, sets the repo as `WorkingDirectory`, pins `HOME`/`AIUW_CONFIG_DIR` to the current user, and uses a stable local/system `PATH`, so the service log does not contain a QR code, pairing payload, bearer token, snapshots, provider output, or transient shell paths. Pair separately with `npm run pair:manual` or by running the connector interactively when you need a QR.

After installing, load it:

```sh
npm run connector:service:load
```

Then check launchd state:

```sh
npm run connector:service:status
```

And verify connector reachability/auth separately:

```sh
npm run connector:status:live
```

`connector:service:status` checks launchd only. `connector:status:live` checks authenticated `/v1/ping` and `/v1/snapshot` on the connector's advertised local addresses and reports only address counts plus sanitized error buckets. A healthy iPhone test needs the second check, even if you are not using the LaunchAgent yet.

Stop it without removing the plist:

```sh
npm run connector:service:unload
```

Remove the plist with:

```sh
npm run connector:service:uninstall
```

Use `npm run connector:service:install:codexbar` only after you intentionally opt into local CodexBar usage parsing.

## Optional Claude Code TTY Usage Adapter

Claude Code's `/usage` screen can expose the current-session and weekly usage bars after the TTY has had a few seconds to render. The connector can read those bars through an explicit opt-in local TTY adapter:

```sh
npm run connector:service:install:claude-code-tty
```

This is intentionally not the default. It starts Claude Code in a detached local terminal session, sends `/usage`, waits for the usage bars, parses only the two used percentages, reset captions, and sanitized Claude Code version, then converts used percent to remaining percent for the widget. It does not send raw terminal output, account identifiers, prompts, transcripts, model breakdowns, tokens, cookies, or long provider text to the iPhone. If the screen shape changes or the bars do not render, Claude degrades back to `用量装神秘🙄` instead of showing fake percentages.

## iPhone Device Test

Quickly see the next real-device gate with:

```sh
npm run device:next
```

While signing or device visibility is blocked, keep local readiness green in this order:

```sh
npm run device:preflight:local
npm run device:build:unsigned
npm run device:evidence:local
```

`device:preflight:local` is the no-signing readiness proof. The unsigned build proves arm64 iphoneos compilation only; it is not installable proof. `device:evidence:local` confirms source/build wiring that can be checked from the Mac and writes the token-free `00-local-evidence-summary.json` audit snapshot while treating missing Apple signing as a warning. It does not bind screenshots.

After signing, install, and screenshot capture, run strict evidence:

```sh
npm run device:evidence
```

Strict `device:evidence` may still report the expected signing failure until signing is configured and remains the post-install screenshot proof gate. After install proof exists, strict `npm run device:evidence` exits nonzero until all required screenshots are present and sidecar-bound.

5. Run `npm run device:signing`.
6. For the first signed install, use `npm run device:install -- --allow-provisioning-updates` if Xcode needs to register the device or refresh profiles.

3. Make sure `张凯的iPhone` is connected, unlocked, trusted, and online in Xcode.
